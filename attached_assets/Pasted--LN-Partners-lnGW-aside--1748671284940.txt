# LN Partners 그룹웨어(lnGW) 시스템 기획서

## 시스템 개요 요약

<aside>
**목적:** 고객, 제품, 매출, 영업 데이터를 단일 플랫폼에 통합하여 실시간 의사결정을 지원합니다. 경영진과 팀원들이 필요한 정보를 신속하게 파악하고 대응할 수 있도록 합니다.

</aside>

- **플랫폼:** PC 웹 브라우저 기반 시스템이며, 모바일 접근성을 위한 반응형 UI를 제공합니다.
- **기술 스택:**
    - 라이브러리(프레임워크) : Javascript 기반 Next.js(app router이용) / Typescript 사용안함
    - 프론트엔드 :
        - Next.js(App Router)와 TailwindCSS,
    - 백엔드 :
        - Node.js(Express)와 MySQL 기반 GraphQL API 서버
    - 상태 관리는 Redux(툴킷)와 redux-persist를 활용하며, JavaScript로 개발됩니다.
- **배포 환경:** 사내 서버 또는 Cloudflare Tunnel 기반 클라우드 사설망에서 운영되며, Docker 컨테이너로 일관된 배포 환경을 유지합니다.
- **언어 지원:** 한국어를 기본으로 제공하며, 향후 베트남어와 영어 지원이 가능한 다국어 구조를 갖춥니다.
- **권한 관리:** 역할 기반 접근 제어(RBAC)로 사용자 역할(관리자, 영업, 마케팅, CS 등)에 따른 메뉴 접근과 기능 사용을 제어합니다.

## 주요 기능 모듈

### 1. 고객관리 (CRM)

고객관리 모듈은 고객 정보와 관계 현황을 관리하는 CRM 기능을 제공합니다. 영업/CS 담당자는 고객 등급, 만족도 지표, VOC(Voice of Customer) 내용을 기록하고, 과거 활동 이력(문의, 미팅, 지원 내역 등)을 조회할 수 있습니다. 이를 통해 고객별 맞춤 전략 수립과 이탈 방지에 활용합니다.

<aside>
예시 사용자 흐름: 영업 담당자가 신규 고객을 등록하고 방문 미팅 결과를 기록하는 과정

</aside>

- ① **신규 고객 등록:** 고객사별 **담당자**(연락처) 정보를 다중 관리. 주요 연락처의 이름, 직책, 전화, 이메일 등을 저장하고 필요시 업데이트.
- ② **VOC 등록:** CS 담당자가 고객 불만사항이나 요청 사항을 접수하여 고객 프로필의 VOC 섹션에 내용을 기록합니다.
- **고객 분류 및 등급**: 고객의 **등급**(예: A/B/C 등급 또는 VIP 표시)을 설정하여 분류. 등급에 따라 차별화된 영업 전략을 적용하고 관리.
- **고객 이력 관리**: 고객과의 **상담/접촉 이력** 기록. 통화, 미팅 내용과 같은 메모를 남기고 관련 파일(견적서 등)을 첨부하여 고객사별 히스토리를 관리.
- ④ **CRM 조회:** 이후 영업/마케팅 팀원은 해당 고객 카드에서 등급, 최근 VOC, 지난 활동 이력을 실시간으로 조회하고 대응 전략을 논의합니다.
- **검색 및 필터**: 등록된 고객사를 **이름, 업종, 등급** 등 조건으로 검색하거나 필터링하여 원하는 고객 정보를 빠르게 조회.
- 입력 및 출력 데이터: 고객관리 기능에서 사용자는 고객 정보를 입력하고, 시스템은 다음과 같은 데이터를 저장/제공합니다.
    - **입력:** 고객 기본정보(회사/이름, 연락처, 주소 등), 고객 등급/분류, VOC 내용, 미팅/활동 메모 등.
    - **출력:** 고객 상세 프로필(기본정보, 등급, 누적 매출 등), VOC 히스토리, 활동 이력 타임라인, 고객 만족도 지표 등이 화면에 표시됩니다.

### 2. 영업관리 (Sales)

영업관리 모듈은 영업 파이프라인과 성과 지표(KPI)를 관리합니다. 영업 팀장은 잠재 고객과 진행 중인 딜(Deal)의 단계별 현황을 파악하고, 팀원별 영업 실적을 모니터링할 수 있습니다. 또한 미팅/전화 기록을 통해 어떤 활동들이 이루어졌는지 추적합니다.

<aside>
예시 사용자 흐름: 영업 팀원이 새로운 영업 기회를 파이프라인에 추가하고 진행 상황을 업데이트하는 경우

</aside>

- **영업기회(리드) 등록**: 잠재 고객 **영업기회**(리드)를 신규 등록. 거래명, 예상 거래금액, 예상 **종료일**(클로즈 날짜) 등 정보를 입력.
- **미팅 기록:** 해당 딜과 관련된 미팅 일정을 추가하고, 일정이 완료되면 결과를 기록합니다 (예: 고객 요구사항 확인, 다음 액션 설정).
- **견적 및 제안서 관리**: 영업기회와 연계된 **견적서** 작성 및 관리. **제품 목록**에서 품목을 선택하면 가격이 자동 계산되어 견적서 생성. 생성된 견적서는 PDF로 **내보내기**하거나 고객에게 **공유** 가능.
- **단계 업데이트:** 시간이 지나며 딜의 진행 상황이 변하면 (예: 견적 제출 → 협상 단계), 담당자가 파이프라인 단계 정보를 업데이트합니다.
- **영업 활동 관리**: 영업과 관련된 전화, 방문, 이메일 등의 **활동 일정**을 기록/관리. **캘린더**와 연동되어 영업 일정에 대한 알림 제공. 활동 결과나 메모를 영업기회에 남겨 팀원들과 정보 공유
- **KPI 대시보드 확인:** 영업 팀장이나 관리자가 시스템의 KPI 대시보드 화면에서 팀 전체의 파이프라인 현황, 단계별 딜 수, 개인별 실적(예: 월별 신규 딜 수, 달성률 등)을 실시간 확인합니다.
- 입력 및 출력 데이터: 영업관리에서 사용되는 주요 입력/출력 데이터는 다음과 같습니다.
    - **입력:** 딜 정보(고객, 관심 제품, 예상 금액 등), 현재 영업 단계, 미팅/통화 메모, 일정 일정(캘린더) 등.
    - **출력:** 영업 파이프라인 보드(단계별 딜 목록), 개별 딜의 상세 정보, KPI 대시보드 차트(팀 실적, 목표 대비 달성률), 활동 보고서 등이 제공됩니다.

### 3. 제품관리 (Product)

제품관리 모듈에서는 회사가 취급하는 제품 목록과 상세 정보를 관리합니다. 마케팅 담당자는 제품의 모델 구조(제품 계층, 규격 등)와 경쟁사 제품 대비 가격 정보를 업데이트하여, 영업팀이 제안서 작성 시 참고할 수 있게 합니다. 또한 제품별 매출 통계를 확인하여 어떤 제품이 인기 있는지 파악할 수 있습니다.

<aside>
예시 사용자 흐름: 마케팅 담당자가 신제품을 등록하고 경쟁사 정보를 업데이트하는 경우

</aside>

- **제품 관리**: 판매하는 **제품 목록**을 시스템에 등록/관리. 제품명, 제품 코드, **가격**, 규격, 설명 등을 입력하고 수정
    - 제품목록 : 제품 등록 시 제품 별 다수의 모델 등록지원
        - 제품 별 모델목록 : ex) Elasty(제품명) , E, G, D(모델명)
    - 제품 별 재고 추가(현재 수량, 판매수량, 샘플 수량, 불량/교환 수량 등)
- **신제품 등록:** 신제품 출시 시, 제품명, 카테고리, 규격, 기본 가격 등의 정보를 입력하여 시스템에 제품을 추가합니다.
- **제품 분류**: 제품을 **카테고리**별로 분류하여 관리 (예: 제품군 등). 카테고리별로 판매 현황 **분석**이 가능하도록 데이터 제공.
- **제품 검색 및 필터**: 제품명을 키워드로 **검색**하거나 카테고리, 가격 범위 등으로 **필터링**하여 원하는 제품 정보를 신속히 조회.
- **가격 책정 및 할인**: 제품별 **가격 정책**(예: 정가, 할인가, 대량구매 할인)을 설정 관리. 영업관리 모듈에서 견적 시 설정된 가격 정책을 자동 적용하여 일관성 유지.
- **경쟁사 가격 입력:** 해당 제품과 유사한 경쟁사 제품들을 등록하고 그 가격 정보를 기록해둡니다.
- **연동된 매출 통계 확인:** 영업팀은 매출관리 모듈과 연계되어 제품별 판매량, 매출 금액 통계를 조회합니다 (예: 분기별 제품 매출 순위).
- **제품 정보 활용:** 영업 사원이 고객사에 제안서를 작성하거나 문의 응대 시, 시스템의 제품 관리 화면에서 최신 제품 스펙과 가격 (및 경쟁사 대비 장점) 정보를 확인하여 활용합니다.
- 입력 및 출력 데이터: 제품관리 기능을 통해 다루는 데이터는 아래와 같습니다.
    - **입력:** 제품 정보(제품명, 코드, 사양, 가격), 제품 분류 체계(모델 계층 구조), 경쟁사 제품명 및 가격, 관련 메모 등.
    - **출력:** 제품 목록 및 검색 화면, 선택된 제품의 상세 정보 페이지(사양, 가격, 관련 문서), 제품별 매출 통계 차트 혹은 표 등이 표시됩니다.

### 4. 매출관리 (Sales)

매출관리 모듈은 견적서 작성부터 수주, 매출 기록까지 영업 프로세스의 금전적 부분을 관리합니다. 영업 담당자는 고객에게 보낼 견적서를 시스템에서 직접 작성하고, 수주 확정 시 발주/매출로 전환하여 기록할 수 있습니다. 또한 정산 내역(입금 확인, 커미션 등)과 누적 매출을 관리하며, 기존에 Google 스프레드시트로 관리되던 매출 데이터를 이관/통합하는 것도 고려합니다.

<aside>
예시 사용자 흐름: 영업 담당자가 견적서를 작성하여 발주를 확정하고 매출을 기록하는 과정:

</aside>

- **견적 생성:** 시스템에서 고객, 제품, 수량, 할인 등을 선택/입력하여 공식 견적서를 생성하고 PDF로 출력하거나 이메일로 전송합니다.
- **발주 확정:** 고객이 견적에 동의하면, 해당 견적 레코드를 "수주 확정" 상태로 변경하고 발주 정보를 추가로 입력합니다 (예: 발주 번호, 납기일 등).
- **매출 기록:** 제품/서비스가 제공되면, 해당 건을 매출 확정으로 표시하고 입금일, 금액을 기록합니다. 매출 기록 페이지는 구글 스프레드시트(엑셀 형태)로 구현합니다.
    - (시트 헤더 항목)날짜 , 고객사, 품목, 제품명, 수량, 소비자가, 단가, 합계, 고객사 결제여부, 결제일, 할인율, 인센티브A, 인센티브B, 배송비, 기타비용, 원가, 마진, 원가합, 마진합, 마진요율, 비고
- **매출 통계 확인:** 관리자는 월별/분기별 매출 현황, 목표 대비 실적을 시스템 내 통계 화면이나 리포트로 확인합니다. (초기 도입 시 기존 스프레드시트에 있던 과거 매출 데이터도 불러와 누적 통계에 반영)
- **입금 및 미수금 관리**: 매출에 대한 **입금** 확인 및 미수금(미결제 금액) **추적**. ⚠️ 일정 기간 미수금 발생 시 영업 담당자에게 자동 알림을 보내 후속 조치
- **목표 대비 실적 관리**: 영업팀의 **연간 매출 목표** 대비 현재 **달성률**을 계산/표시. 대시보드에 ✅ **목표 달성 여부**를 표시하여 목표 대비 진행 상황을 시각화.
- **데이터 공유 및 내보내기**: 매출 데이터를 Excel/CSV 등으로 **내보내기**하여 재무팀과 정보 공유. 내보낸 자료를 통해 **회계 처리**나 추가 분석을 진행.
- 입력 및 출력 데이터: 매출관리 모듈에서 다루는 주요 데이터 요소:
    - **입력:** 견적 품목 목록(제품, 수량, 단가), 견적서 메타데이터(견적 번호, 날짜, 유효기간 등), 발주 정보(발주일, 납기), 매출 정보(매출일, 입금액, 결제 방식), 정산 메모 등.
    - **출력:** 견적서 문서(PDF 등), 개별 거래(딜)의 진행 상태(견적 → 발주 → 매출) 표시, 매출 목록 및 상세 내역, 기간별 매출 집계표/차트, 미수금/정산 상태 표시 등이 제공됩니다.

### 5. 전자결재 (Approval)

전자결재 모듈은 회사 내부의 각종 결재 프로세스를 온라인화합니다. 사용자들은 출장보고, 계약서 검토, 비용 승인 등 문서를 등록하고 결재선을 지정하여 결재를 상신합니다. 결재 문서는 지정된 순서에 따라 승인 혹은 반려되며, 각 단계에서 관련자에게 알림(이메일 또는 앱 알림)이 전송됩니다.

<aside>
예시 사용자 흐름: 직원이 내부 결재 문서를 작성하여 결재를 완료받는 과정:

</aside>

- **결재 문서 작성**: 품의서, 지출결의서 등 **결재 문서**를 작성하여 상신. 문서 양식 템플릿을 제공하여 편리하게 작성하고 필요한 경우 관련 **근거 파일**을 첨부.
- **결재선 지정 및 진행**: 결재 상신 시 미리 정의된 **결재선**(결재자 순서)을 선택 또는 지정. 예: 팀장 → 본부장 → 임원 순의 다단계 결재 진행. 결재 진행 상황을 실시간 **추적**하여 요청자와 결재자에게 알림.
- **결재 상신:** 작성된 문서를 제출하면, 첫 번째 결재자에게 알림이 가고 시스템 내 결재함에 해당 문서가 대기 상태로 표시됩니다.
- **결재진행(승인/반려 처리)**: 결재 요청에 대해 **승인** 혹은 **반려** 처리. 결재자는 의견을 첨부할 수 있으며, 🔄 반려 시 요청자는 내용을 수정 후 재상신 가능. 승인 완료 시 신청자에게 결과가 통보됩니다. 모든 결재선의 승인이 완료되면 최종 승인으로 문서 상태가 갱신됩니다.
- **완료 및 보관:** 결재 완료된 문서는 관련자들에게 완료 알림이 가며, 문서 원본과 승인 이력은 전자결재 모듈 내 문서함(보관함)에 저장되어 추후 열람할 수 있습니다. 완료된 결재 문서는 **문서함**에 자동 보관. 결재 번호, 제목, 결재자 등으로 **검색** 및 필터링하여 과거 결재 기록을 조회.

**입력 및 출력 데이터:** 전자결재 기능을 통해 다루는 데이터:

- **입력:** 결재 문서 종류/양식, 본문 내용, 첨부 파일, 지정된 결재자 목록, 코멘트(결재자의 의견) 등.
- **출력:** 결재 문서 상세 보기(본문, 첨부, 현재 결재 진행 상태), 결재함 목록(대기/진행/완료 문서 목록), 이메일/애플리케이션 알림 (결재 요청, 승인/반려 결과), 최종 승인된 문서 보관함 등이 제공됩니다.

### 6. 기타 관리 기능

기타 부가적인 관리 기능으로 문서 저장소, 시스템 로그, 재고 관리 연동 등을 포함합니다. 이들 기능은 핵심 업무 기능은 아니지만 그룹웨어의 완성도와 확장성을 높여 줍니다.

- **📁 문서 저장소**
    
    영업 계약서, NDA, 제품 메뉴얼 등의 파일을 중앙에 보관하는 저장소입니다. 사용자들은 권한에 따라 문서를 업로드/다운로드할 수 있으며, 버전 관리나 만료일 관리 기능을 통해 최신 문서와 보안을 유지합니다. 예를 들어, 영업팀은 계약서를 이곳에 업로드하여 필요한 부서가 언제든 열람할 수 있습니다.
    
- **📊 시스템 로그**
    
    사용자의 주요 행동(로그인, 데이터 수정/삭제 등)을 기록하는 활동 로그입니다. 관리자는 로그를 통해 어떤 사용자가 언제 어떤 작업을 했는지 모니터링하여 보안 감사나 문제 발생 시 추적할 수 있습니다. (예: 관리자 화면에서 사용자별 최근 접속 시간, 데이터 변경 이력 조회 가능)
    
- **👥 사용자 및 권한 관리**: 시스템 **사용자 계정** 생성/삭제 및 권한 부여/회수 관리. 부서, 직급에 따른 **역할(Role)**을 정의하고 모듈별 **접근 권한**을 설정. 관리자 화면에서 사용자 목록과 권한을 일괄 관리.
- **📢 공지사항 / 게시판**: 회사 내부 **공지사항**을 게시하고 구성원들에게 전달. 부서별/프로젝트별 **게시판**을 제공하여 자료 공유 및 토론. 새로운 공지 작성 시 전 직원에게 ✅ **알림** 전송.
- **📅 일정 / 캘린더**: 개인 및 팀 **일정 관리** 기능. 영업 일정, 결재 일정 등을 통합 **캘린더**에서 확인. Outlook  또는 Google Calendar와 🔄 **양방향 동기화** 옵션을 제공하여 외부 일정과 연동.
- **🔔 알림 및 검색**: 주요 이벤트 발생 시 실시간 **알림**(웹 알림, 이메일 또는 모바일 푸시) 제공. 시스템 내 모든 모듈 데이터를 대상으로 하는 **통합 검색** 기능으로 고객, 영업, 결재 문서를 빠르게 검색.
    
    

## UI 예시 구성

다음은 예상되는 주요 화면의 예시입니다. 각 화면은 사용자 역할과 시나리오에 맞춰 직관적인 UI로 설계됩니다.

### 고객사 상세 화면

선택한 고객의 상세 정보를 보여주는 화면입니다. 상단에는 고객 기본 프로필(회사명, 담당자, 등급, 만족도 지표)이 표시되고, 하단에는 해당 고객과 관련된 최근 활동 이력 목록과 열린 영업 기회가 나타납니다. 또한 "VOC 추가"나 "미팅 일정 추가" 등의 액션 버튼을 통해 바로 고객 관련 활동을 기록할 수 있습니다.

### 영업 프로세스 단계

1. ➤ **신규 고객 등록:** 영업사원은 **고객관리** 모듈에서 신규 고객사를 등록합니다. 고객사 이름, 업종, 연락처 등을 입력하고 ✅ **저장**합니다. 등록 완료 시 해당 고객사가 고객 리스트에 추가됩니다.
2. ➤ **영업기회 생성:** 방금 등록한 고객사에 대해 **영업관리** 모듈에서 새로운 영업기회를 생성합니다. 거래명, 예상 매출액, 예상 **종료일** 등을 입력하고 관련 **제품**을 선택합니다. 생성된 영업기회는 영업 파이프라인의 **초기 단계**로 등록됩니다.
3. ➤ **영업 활동 진행:** 영업 담당자는 영업기회에 대해 고객과 상담을 진행하며 **영업 단계**를 진척시킵니다. 접촉 단계에서 제안, 협상 단계로 영업기회를 **업데이트**하고, 주요 진행 사항을 메모로 기록합니다. ⚠️ 예상보다 큰 할인 등 특별 조건이 필요하면 해당 내용을 영업기회 메모에 기재합니다.
4. ➤ **견적서 발행 및 결재:** 협상이 완료되면 **제품관리** 모듈의 가격 정보와 연동하여 견적서를 발행합니다. 할인율이 큰 경우 사전에 정해둔 기준에 따라 **전자결재** 모듈로 할인 **승인 결재**를 올립니다 (예: 팀장 승인이 필요한 할인율 초과 시). 팀장은 결재 문서를 검토하여 ❗ 승인 또는 반려합니다. 🔄 반려 시 영업담당자는 견적 내용을 수정한 후 재상신합니다.
5. ➤ **계약 체결 및 매출 등록:** 고객이 견적 조건에 동의하여 **계약 체결**이 완료되면, 해당 영업기회를 **성공 (Win)** 상태로 마감합니다. 동시에 **매출관리** 모듈에 계약 내용이 매출 기록으로 등록되어 **청구서**가 발행됩니다. 매출 기록에는 계약 금액, 판매 제품, 고객사 정보가 연동되어 저장됩니다.
6. ➤ **사후 처리 및 공유:** 매출 등록 후 영업기회는 완료 처리되고, 관련 데이터가 매출 통계에 반영됩니다. 영업팀 관리자는 대시보드에서 최신 **매출 현황**과 **영업 실적**을 확인합니다. 또한 관리자는 **공지사항**을 통해 신규 계약 성사를 팀 전체에 공유하고, 성과를 ✅ **격려**합니다.

## 데이터베이스 모델링 제안

주요 엔티티에 대해 다음과 같이 테이블 구조와 관계를 설계할 수 있습니다:

| 테이블 | 설명 (주요 컬럼) | 관계 (Relationship) |
| --- | --- | --- |
| User (사용자) | 시스템 사용자 계정 정보 (사용자 ID, 이름, 부서, 이메일 등) | 한 사용자 → 여러 Sales, 여러 Log |
|  |  | 한 사용자 → 여러 Approval (결재 문서 작성자) |
| Role (역할) | 역할 정의 (역할ID, 이름, 설명) | 한 역할 → 여러 User (RBAC 적용 위해 다대다, UserRole 매핑) |
| UserRole (사용자-역할) | 사용자와 역할의 매핑 (UserID, RoleID) | (복합 PK, 다대다 관계 구현용 중간 테이블) |
| Customer (고객) | 고객사 및 고객 정보 (고객ID, 회사명, 담당자, 연락처, 등급 등) | 한 고객 → 여러 Sales, 여러 VOC/Activity |
| Product (제품) | 제품 정보 (제품ID, 이름, 분류, 가격, 사양 등) | 한 제품 → 여러 Sales (여러 매출 항목에 사용) |
| Sales (매출/딜) | 영업 기회 및 매출 거래 (거래ID, 고객ID, 담당자ID, 제품ID, 단계, 금액, 상태 등) | 여러 SalesItem 구성 가능 (견적 품목별 상세) |
|  |  | 여러 Sales → 하나 Customer, 하나 User (담당자) |
| SalesItem (거래품목) | 개별 거래에 연결된 품목 목록 (SalesID, ProductID, 수량, 가격 등) | 한 Sales → 여러 SalesItem (예: 견적서 내 여러 제품 라인) |
| VOC/Activity (고객활동) | 고객 VOC 및 활동 이력 (활동ID, 고객ID, 유형, 내용, 담당자ID, 날짜 등) | 한 Customer → 여러 VOC/Activity 기록 |
|  |  | 한 User → 여러 VOC/Activity 기록 |
| Approval (결재문서) | 전자결재 문서 (문서ID, 종류, 작성자ID, 상태, 생성일 등) | 한 Approval → 여러 ApprovalStep |
|  |  | 한 User → 여러 Approval (작성 또는 승인) |
| ApprovalStep (결재선) | 결재 단계 (StepID, 문서ID, 결재자ID, 순서, 상태 등) | 한 Approval → 여러 ApprovalStep (결재자 순서대로 연결) |
| Log (시스템로그) | 사용자 활동 로그 (로그ID, 사용자ID, 활동종류, 내용, 타임스탬프) | 한 User → 여러 Log |
| Document (문서저장소) | 업로드한 파일 정보 (문서ID, 파일명, 경로, 업로드한 사용자ID, 분류 등) | 한 User → 여러 Document (업로더 기준) |
|  |  | 관련 업무 모듈과 링크 (예: 계약서 → SalesID로 연계) |
| Inventory (재고) | 제품 재고 현황 (제품ID, 창고 위치, 재고 수량 등) – 외부 연동 | 한 Product → 하나 Inventory 레코드 (1:1 또는 지점별 1:다 구조) |

관계 요약: 고객-영업(매출) 간 1대다, 제품-매출(거래 품목) 간 1대다 관계입니다. 사용자와 역할은 다대다 관계이며, 전자결재 문서는 여러 결재 단계(ApprovalStep)를 가집니다. 문서 저장소와 재고는 다른 엔티티와 느슨하게 연계되며, 필요 시 키로 참조하도록 설계합니다 (예: Document 테이블에 SalesID 컬럼을 두어 특정 매출건의 계약서를 연결)
